## Context

当前仓库已经有一份 HTML 形式的项目设计文档，但还没有可直接执行的 OpenSpec 变更产物。用户已经补齐了本轮最关键的架构决策：CLI 框架固定为 `click`，默认输出固定为 `json`，二进制能力必须真正支持二进制读写与处理，工具还必须支持批量执行。项目运行环境已经固定在 `D:\github-project\fileglide\.venv`，因此本次设计不需要再为多解释器环境做抽象。

从系统职责看，`fileglide` 不是一个简单的 shell 包装器，而是面向 AI agent 的本地文件系统能力层。它必须同时覆盖路径与文件生命周期管理、文本和二进制内容读写、精确编辑、遍历、搜索、模糊匹配、编码适配、默认 JSON 输出和批量计划执行。这样的能力面天然跨越多个模块，因此需要先固化分层、契约、风险控制和实施顺序。

## Goals / Non-Goals

**Goals:**
- 建立基于 `click` 的多层命令树，保证命令契约、帮助文本和测试方式稳定。
- 把默认输出定义为 `json`，让 agent 在不额外传参时就能获得稳定结构化结果。
- 同时支持文本和二进制文件，不把二进制对象简单视为“跳过项”。
- 建立遍历、搜索、读写、精确编辑、批量执行和编码服务的清晰分层。
- 设计可直接落地的实施顺序，使后续实现可以按最小闭环逐步推进。

**Non-Goals:**
- 本变更不引入 GUI、不做远程文件系统、不做数据库索引服务。
- 本变更不引入后台常驻守护进程。
- 本变更不把自然语言直接解释成批量任务计划，批量执行仍基于显式计划文件。
- 本变更不在第一阶段引入 AST 级语义重构能力，精确编辑仍以文本与字节定位为主。

## Decisions

### Decision 1: CLI 框架固定使用 `click`
- 选择：使用 `click` 构建根命令、命令组与叶子命令。
- 原因：仓库参考项目 `OpenTrade` 已经验证了 `click` 在多层命令树、帮助信息、命令测试和运行时参数组织上的可维护性。`click` 对批量命令、路径参数、布尔开关和结构化子命令的支持也更成熟。
- 备选方案：`Typer` 更现代，但会在此阶段引入额外风格差异与装饰器约束；`argparse` 过于底层，不适合构建大规模命令树。

### Decision 2: 默认输出固定为 `json`
- 选择：所有命令默认输出 `json`，仅在显式传入其他格式参数时切换到 `jsonl`、`table`、`text` 等扩展格式。
- 原因：用户已经明确指定默认输出为 `json`，而且工具的主要受众是 AI agent。把结构化输出设为默认值，能减少 agent 端的解析歧义，也能让批量执行和错误处理复用统一字段。
- 备选方案：默认 `table` 更利于人工浏览，但会显著增加 agent 场景下的解析成本；默认 `text` 则不具备可靠契约。

### Decision 3: 文本与二进制并行支持，而不是互斥支持
- 选择：将内容操作拆成统一入口下的文本路径与二进制路径，二者共享元数据、路径和执行结果模型，但在读写、切片、检索和编辑行为上有明确分流。
- 原因：用户明确要求“二进制支持二进制”，这意味着系统必须允许二进制读取、写入、复制、大小感知、字节切片和批量处理，不能简单返回“binary skipped”。
- 实现含义：
  - 文本读写走编码探测、解码、编辑、再编码链路。
  - 二进制读写走字节流链路，不做隐式解码。
  - 搜索层区分文本正则搜索和二进制字节搜索。
- 备选方案：只支持文本并拒绝二进制最简单，但与目标冲突。

### Decision 4: 服务分层采用“薄入口 + 命令构建 + 执行器 + 门面 + 领域服务”
- 选择：延续 `OpenTrade` 的骨架思路，按 `main/app -> commands -> executor -> facade -> services` 分层。
- 原因：这可以把 CLI 参数解析、运行时控制、领域语义和底层 I/O 解耦，避免未来在支持批量执行、编码策略和二进制操作时把逻辑揉进一个巨型命令文件。
- 备选方案：直接在命令回调中调用文件函数实现更快，但后续扩展 JSON 输出、dry-run、批量计划和错误码时会迅速失控。

### Decision 5: 搜索分成文件系统检索与内容检索两条能力线
- 选择：
  - `path/file search` 负责路径、文件名、相对路径匹配。
  - `text grep` 和未来的二进制搜索命令负责内容匹配。
- 原因：路径名匹配和内容匹配的输入、输出、排序逻辑与风险控制完全不同，混在一个命令下会让结果结构难以稳定。
- 备选方案：单一 `search` 命令功能过重，后期难以维护。

### Decision 6: 模糊匹配默认基于 `vortezwohl` 的 Levenshtein 能力
- 选择：文件名和路径名模糊匹配使用 `vortezwohl.nlp.LevenshteinDistance`。
- 原因：这是用户显式要求，同时仓库依赖中已包含 `vortezwohl`。这样可以避免自写低质量模糊排序，并让能力与既有依赖保持一致。
- 备选方案：自定义距离函数实现成本低，但可解释性和稳定性较差。

### Decision 7: 批量执行采用显式计划文件，而不是自然语言
- 选择：为批量执行引入计划文件格式，例如 JSON 计划，内含命令类型、参数、预期结果和 dry-run 控制。
- 原因：批量执行涉及多目标文件和路径时，最重要的是可审计、可重放、可预演。显式计划文件比直接解析自然语言更安全，也更适合 agent 自动生成后再执行。
- 备选方案：命令行连续拼接虽然方便，但不适合复杂批量任务的恢复、验证与失败重试。

### Decision 8: 编码服务独立为核心领域服务
- 选择：把 UTF-8 无 BOM 优先、BOM 识别、常见编码候选、替换风险提示、写回保真策略放进独立 `EncodingService`。
- 原因：编码兼容是这个项目的主特性之一，不应散落在各命令里。独立服务也更利于做样本测试和后续策略扩展。
- 备选方案：在每个读取命令里临时探测编码会造成逻辑重复，且难以保证写回一致性。

### Decision 9: 统一结果模型与错误模型
- 选择：所有命令返回统一结构骨架，至少包含 `ok`、`operation`、`targets`、`result`、`errors`、`warnings`、`meta` 字段。
- 原因：默认 JSON 输出只有在字段稳定时才真正有价值。批量执行、dry-run、二进制操作和搜索结果也都需要共享同一层结果契约。
- 备选方案：各命令各自输出自由 JSON，短期快，长期不可维护。

## Risks / Trade-offs

- [风险] 默认 JSON 会让纯人工用户感觉不够友好 → 缓解：保留显式 `--format table|text|jsonl`，但不改变默认值。
- [风险] 二进制支持扩大问题域，测试量明显上升 → 缓解：先支持二进制读取、写入、复制、切片与大小感知，复杂字节搜索和批量二进制修改放到后续阶段。
- [风险] 编码“完美适配”在现实中存在探测歧义 → 缓解：把“可诊断、可回退、可提示”写进契约，避免虚假承诺 100% 自动正确。
- [风险] 批量执行可能放大误操作 → 缓解：计划文件、默认 dry-run、显式确认、结果摘要和失败中止策略必须先于真实 apply 落地。
- [风险] Click 命令树在第一阶段搭太大，会拖慢最小闭环 → 缓解：先只实现核心命令骨架，其他命令保留占位但不提前实现业务逻辑。

## Migration Plan

1. 先搭建 `fileglide` 包骨架与 `click` 根命令入口。
2. 先实现统一 JSON 结果模型、错误模型与基础路径/文件元数据结构。
3. 先完成文件/路径创建删除、基础遍历、文本读取和文本覆盖写入的最小闭环。
4. 再引入 `EncodingService`、搜索服务和 `vortezwohl` Levenshtein 模糊匹配。
5. 再扩展二进制读写、二进制切片和批量计划执行。
6. 最后补充精确编辑、批量执行回滚策略与更丰富的格式输出。

回滚策略：
- 在实现阶段，每个能力按独立任务切片推进，保证失败时可只回滚对应服务与命令。
- 批量执行与 destructive 操作必须在 dry-run 与测试通过前保持默认安全状态，不直接开放激进行为。

## Open Questions

- 批量计划文件的首选格式是否固定为 JSON，还是同时支持 YAML。
- 二进制搜索第一版是否需要支持十六进制模式匹配，还是先只支持字节序列精确匹配。
- 默认 JSON 输出中，是否为大型内容字段提供截断与 `--full` 解锁机制。
- Windows 长路径、符号链接和权限异常在第一阶段需要覆盖到什么程度。
